Quickstart

Get up and running with Hylé in under 5 minutes.

1. Install Hylé (macOS detected)

brew install hyle
# or
npm install --global hyle

2. Pull a substrate

cd my-project/
hyle pull hyle-org/starter

3. Start working

hyle prompt "generate a REST CRUD boilerplate"
hyle done
Use hyle prompt --dry-run to preview what context the Epitomator will inject before running.

Configuration

All configuration lives in .hyle/hyle.yaml at the root of your project.

# .hyle/hyle.yaml
substrate:
  name: my-project
  version: 1.0.0
  registry: hyle-org/starter

budget:
  max_tokens_per_session: 50000
  warn_at: 80

scopes:
  - id: public
    description: All non-sensitive documents
  - id: internal
    description: Internal specs only

governance:
  ethics_path: ethics/policies.yaml
  strict: true

Key fields

FieldDescription
substrate.nameProject identifier used in audit logs
budget.max_tokens_per_sessionHard cap on token usage per hyle prompt call
governance.strictExit non-zero on any ethics violation
scopesNamed data access perimeters for agents

Best Practices

Structure your ontology first

Before prompting agents, ensure ontology/ contains high-weight anchor documents. The Epitomator weights documents during injection — sparse ontologies lead to unfocused output.

Use scopes for data isolation

hyle scope add internal ontology/specs/secret-api.md
hyle prompt "generate client code" --scope public

Run governance checks in CI

hyle scope check all --strict
hyle topology check
hyle substrate check-anchors --strict

Tag before publishing

Always run hyle substrate tag <version> before publishing. Tags freeze model weights and create a reproducibility snapshot, allowing other users to replay prompts with identical conditions.

Use hyle substrate reproduce <tag> "your prompt" to verify reproducibility before releasing a new version.

Publishing a Print

Anyone can publish a print to the Hylé Prints registry. Prints are backed by public GitHub repositories.

1. Author your substrate

mkdir my-print && cd my-print
hyle init --yes
# populate ontology/, identities/, craft/, ethics/

2. Test it

hyle substrate test
hyle substrate check-anchors --strict
hyle governance check "scaffold a feature"

3. Tag and publish

hyle substrate tag 1.0.0
hyle substrate publish myuser/my-print

4. Push new versions

# after making changes:
hyle substrate tag 1.1.0
hyle substrate publish myuser/my-print
The registry links directly to the GitHub repository. Ensure your repo is public before publishing.

Philosophy

Hylé (from the Greek ὕλη, matter or material) is built on the idea that AI agents should work within structure, not instead of it.

Substrates over prompts

A one-off prompt is ephemeral. A substrate is persistent, versioned, and reviewable. Hylé treats the agent's working context as a first-class artifact — one that can be audited, diffed, and reproduced.

The four folders

Every print is structured around four orthogonal concerns: ontology (what the agent knows), identities (who the agent is), craft (how the agent works), and ethics (what the agent must never do). This separation makes prints composable and auditable.

Reproducibility

Model outputs are non-deterministic, but the context fed to the model need not be. Hylé's weight system and snapshot tags allow any prompt to be replayed under identical conditions, making AI-assisted development auditable in regulated environments.
