Quickstart

Get up and running with Hylé in under 5 minutes.

1. Install Hylé

# macOS
brew install hyle

# Windows
choco install hyle

# Linux
curl -fsSL https://get.hylé.com | sh
# or: cargo install hyle-cli

2. Pull a community blueprint

hyle search java spring tdd
hyle pull claude-java-springboot

3. Publish your own

cd my-project/
hyle init
hyle ontology   # scan for CLAUDE.md, specs, diagrams…
hyle craft      # scan for ARCHITECTURE.md, package.json…
hyle push --new my-blueprint   # publish as me/my-blueprint@0.1.0
Core CLI is programmatic only — no LLM required, no network call to an AI on every command. Fast and scriptable.

Configuration

Hylé uses a two-layer config system: global defaults at ~/.hyle, local overrides at <project>/.hyle. Local values take precedence field by field.

# ~/.hyle or <project>/.hyle

remote_url: https://registry.hylé.com    # Default registry
currency: EUR                            # Cost estimates in hyle watch (EUR or USD)
default_llm: fallback                    # Model key from hyle.yaml for extensions
auto_inject: true                        # Inject file refs into agent instruction file on pull
contribute_deps: true                    # Share resolved dep install commands with registry

split_threshold: "80%"                   # hyle watch --split threshold (% or abs token count)
split_action: ask                        # ask | link | clipboard
audit_retention_days: 90                 # Days to keep hyle-audit-*.log files

scan:
  ontology: [.md, .pdf, .docx, .feature, .csv, .ts]
  craft: [package.json, pom.xml, angular.json, .md]
  identities: [.md]
  ethics: [.cedar, trulens.yaml, ragas.yaml, guardrails.yaml]

Key fields

FieldDescription
remote_urlRegistry endpoint — default is registry.hylé.com; set a public GitHub URL for a custom remote
auto_injectAutomatically add pulled file references to your agent instruction file (CLAUDE.md, .cursorrules, etc.) when line is not already present
contribute_depsShare resolved dependency install commands back to the registry so future pullers benefit
split_thresholdToken threshold for hyle watch --split: percentage of model context limit or absolute count
audit_retention_daysHow long hyle-audit-*.log files are kept (default 90 days)

Best Practices

Scan before publishing

Run all four scan commands before your first push. Hylé auto-detects common files; use the optional path argument to cherry-pick extras.

hyle ontology    # CLAUDE.md, specs, features, diagrams
hyle craft       # ARCHITECTURE.md, MCP configs, package.json
hyle identities  # AGENTS.md, .claude/agents/*.md
hyle ethics      # *.cedar policies, TruLens, Ragas configs

Use .hyleignore to protect secrets

# .hyleignore
.env
*.pem
*.key
secrets/
config/local.*

Use snapshots for WIP, push for stable

hyle snapshot    # patch bump, not listed as stable — low commitment
hyle push        # minor bump, listed as stable
hyle release     # major bump, breaking changes

Pin models for reproducibility

Declare model_pin in your blueprint's hyle.yaml to freeze the exact model checkpoint. Hylé emails you monthly when a newer checkpoint is available — with a one-click update link.

Provide dependency URLs, not raw install commands

Set the url field to the official GitHub repo or package page. Hylé resolves the correct install command per OS automatically — a macOS user never sees an apt command. Only override with install.macos / install.linux if auto-detection fails.

Publishing a Blueprint

Anyone can publish a blueprint to the Hylé registry. The default registry is registry.hylé.com.

1. Initialize Hylé in your project

cd my-project/
hyle init

Generates a hyle.yaml manifest after asking basic questions. The name + author combination must be unique on the registry — Hylé checks this during init.

2. Scan your project files

hyle ontology    # Finds CLAUDE.md, .cursorrules, specs, features, diagrams
hyle craft       # Finds ARCHITECTURE.md, MCP configs, package.json, pom.xml
hyle identities  # Finds AGENTS.md, .claude/agents/*.md
hyle ethics      # Finds *.cedar policies, TruLens configs, Ragas configs

3. Review and refine hyle.yaml

nano hyle.yaml

Add a .hyleignore to exclude secrets, API keys, and private configs.

4. Publish

hyle push --new my-blueprint   # First publish: creates me/my-blueprint@0.1.0
hyle snapshot                  # Patch bump for WIP — not listed as stable
hyle push                      # Minor bump — listed as stable
hyle release                   # Major bump — breaking changes
On any publish, Hylé bundles all referenced files with a SHA-256 checksum and runs an automatic security scan. Flagged versions are marked [flagged] — not pullable, content hidden. Push history is always public.

Three publish tiers

CommandVersion bumpListed as stableWhen to use
hyle snapshotpatch x.x.+1NoWIP sharing, no SLA
hyle pushminor x.+1.0YesTested, working blueprint
hyle releasemajor +1.0.0YesBreaking changes to structure or file layout

All three accept an explicit version override as an optional argument: hyle push 1.5.0.

Philosophy

Hylé (from the Greek ὕλη, hyle: "wood, matter") is built on the idea that AI agents should work within structure, not instead of it.

Blueprints over one-off prompts

A one-off prompt is ephemeral. A blueprint is persistent, versioned, and reviewable. Hylé treats the agent's working context as a first-class artifact — one that can be audited, diffed, and reproduced.

The four domains

Every blueprint is structured around four orthogonal concerns:

  • ontology — The "what"? Specifications, goals, features, examples and data
  • craft — The "how"? Architecture, technical design, practices and recipes
  • identities — The "who"? Agentic personas, model behaviour specifications
  • ethics — The limits: behaviour constraints, compliance requirements, security concerns, privacy design

The "why"? is between your human hands. This separation makes blueprints composable and auditable.

Programmatic core, optional intelligence

The core CLI requires no LLM — no network call to an AI on every command; fast, scriptable. Optional extensions (hyle watch, hyle index) may use lightweight local or SaaS models, configured in .hyle.

Hylomorphism

In Aristotle's hylomorphism doctrine, ὕλη (hyle: "matter") gains substantial form through act. Projects developed with LLM assistance mimic this potency: LLMs allow humans to digest large amounts of information and arrange its extracted essence into a project form meant to be used in the real world. Hylé helps humans act upon a compound of documents (raw matter) — may it be already structured or spread out and chaotic.
